# 设备场景案例地图教学规格

本文档定义 PixelPipe ISP Lab 中“设备场景案例地图”的教学目标、素材规范和网页实现要求。它不是图库说明，而是讲清楚不同成像设备为什么需要不同 pipeline，以及学习者应该如何通过图像、参数和指标观察差异。

## 1. 设计原则

设备场景地图不能复用普通 ISP demo 缩略图。每张图都必须服务于一个真实成像问题：

- 相机摄影：镜头、景深、RAW 宽容度、色卡校准。
- 手机摄影：小传感器、逆光、高动态范围、多帧 HDR、人像主体保护。
- 电影机：夜景高光、Log/LUT、动态范围、稳定曝光和色彩管理。
- VR 眼镜：左右眼一致性、低延迟、双目标定、畸变和色差校正。
- 医疗成像：灰阶结构、信噪比、结构增强、稳定显示。仅用于教学，不用于诊断。
- 屏幕显示：sRGB/P3/HDR、Gamma/EOTF、Tone Mapping、环境光适配。

每个设备卡片必须回答四个问题：

1. 这张图像对应真实设备中的哪个问题？
2. 它连接到哪个高级成像实验室？
3. 拖动参数时，学习者应该观察画面哪里？
4. 调得过强或过弱会出现什么失败现象？

## 2. 素材来源策略

当前项目使用专门生成的无品牌、无文字、无水印场景图，存放在：

```text
assets/scenarios/
```

原因：

- 网络图片授权不稳定，直接放入开源项目有版权风险。
- 生成图可以准确控制教学元素，例如色卡、逆光窗口、夜景灯牌、双目标定板、灰阶结构和显示校准图。
- 所有图像都能离线运行，GitHub Pages 不依赖外部图片链接。

如果后续替换为真实摄影素材，必须满足：

- 来源页面可公开访问。
- 授权明确，例如 CC0、CC BY、CC BY-SA 或项目可接受的授权。
- 文档中记录作者、文件页、授权和修改方式。
- 不含品牌商标、隐私人物、医疗诊断内容或不可公开数据。

## 3. 六类设备案例

### 3.1 相机摄影

素材文件：

```text
assets/scenarios/camera-photography.png
```

画面元素：

- 无品牌相机和镜头。
- 微距纹理主体。
- 色卡和灰阶块。
- 桌面静物环境。

讲学目标：

- 解释镜头位置如何改变焦平面。
- 解释为什么微距景深浅，自动对焦峰值很窄。
- 解释色卡如何辅助白平衡和 CCM 校准。
- 解释相机 RAW 为什么给后期留出更大调整空间。

网页案例连接：

- 默认连接到“微距纹理对焦”实验室。
- 参数：微距镜头步进。
- 观察：前景纹理、后景纹理、清晰度评分、色卡灰阶中性。
- 失败：镜头步进过快会跳过峰值，低纹理区域会导致拉风箱。

### 3.2 手机摄影

素材文件：

```text
assets/scenarios/phone-photography.png
```

画面元素：

- 无品牌手机。
- 逆光窗口。
- 人像/人形主体。
- 手机屏幕中同时出现高光窗口和暗部主体。

讲学目标：

- 解释手机小传感器为什么更依赖 HDR 和多帧融合。
- 解释逆光场景中人脸/主体测光与高光保护的冲突。
- 解释实时取景器需要低延迟，不能无限堆叠复杂算法。
- 解释肤色保护和局部 Tone Mapping 的作用。

网页案例连接：

- 默认连接到“逆光窗口测光”实验室。
- 参数：曝光 EV。
- 观察：主体亮度、窗外高光、直方图右侧裁剪、暗部噪声。
- 失败：只提高曝光会让窗口过曝；只保高光会让主体变黑。

### 3.3 电影机

素材文件：

```text
assets/scenarios/cinema-camera.png
```

画面元素：

- 无品牌电影机 rig。
- 夜景街道。
- 高亮灯具和深暗阴影。
- 监视器中偏平的 Log-like 预览。

讲学目标：

- 解释电影机为什么强调动态范围和高光 roll-off。
- 解释 Log 曲线如何保留调色空间。
- 解释 LUT 如何把 Log 预览转换为更接近最终观感。
- 解释视频拍摄中曝光变化要平滑，避免闪烁。

网页案例连接：

- 默认连接到“夜景街灯曝光”实验室。
- 参数：夜景曝光 EV。
- 观察：灯牌层次、暗部可读性、画面对比、噪声代价。
- 失败：曝光过高灯具变纯白；曝光过低暗部失去叙事信息。

### 3.4 VR 眼镜

素材文件：

```text
assets/scenarios/vr-headset.png
```

画面元素：

- 无品牌 VR 头显。
- 前置双摄像头区域。
- 棋盘格标定板。
- 左右视角示意卡片。

讲学目标：

- 解释 VR passthrough 为什么需要左右眼曝光、颜色和几何一致。
- 解释双目匹配、视差、深度和投影几何的关系。
- 解释畸变校正、色差校正和显示补偿为什么影响佩戴舒适度。
- 解释低延迟为什么比单帧画质更重要。

网页案例连接：

- 默认连接到“匹配置信度”实验室。
- 参数：匹配置信度。
- 观察：左右结构是否对齐、是否出现重影、匹配线是否可信。
- 失败：错误匹配会造成深度漂移、几何错位和眩晕感。

### 3.5 医疗成像设备

素材文件：

```text
assets/scenarios/medical-imaging.png
```

画面元素：

- 无品牌超声探头。
- 灰阶成像屏幕。
- 结构化 phantom 图样。
- 医学教学场景，而非真实诊断场景。

讲学目标：

- 解释医疗成像为什么重视信噪比、结构边缘和稳定显示。
- 解释去噪不能把弱结构当成噪声抹掉。
- 解释增强算法必须保守、可解释、可回溯。
- 强调本项目只用于图像算法教学，不用于医疗诊断。

网页案例连接：

- 默认连接到“低光运动视频/结构增强”实验室。
- 参数：时域降噪强度。
- 观察：噪声是否减少、结构边缘是否保留、灰阶层次是否稳定。
- 失败：过强去噪会抹掉细结构；过强锐化会制造伪边缘。

### 3.6 屏幕显示效果

素材文件：

```text
assets/scenarios/display-output.png
```

画面元素：

- 无品牌显示器。
- 色块、灰阶、HDR-like 高光和暗部区域。
- 校色器外观元素。
- 环境光源。

讲学目标：

- 解释图像 pipeline 结束不等于显示观感结束。
- 解释 sRGB、Display P3、HDR、Gamma、EOTF 和 Tone Mapping 的关系。
- 解释同一 RGB 数值在不同屏幕、亮度和环境光下观感会变化。
- 解释显示端色彩管理为什么影响手机摄影和相机后期。

网页案例连接：

- 默认连接到“颜色系统与白平衡”实验室。
- 参数：色温。
- 观察：灰阶中性、肤色舒适度、色块饱和度、亮暗部显示。
- 失败：色温过冷或过暖会造成整体偏色；Tone 不当会丢暗部或高光。

## 4. 页面实现要求

设备场景地图的卡片必须包含：

- 专用场景图片。
- 设备名称。
- 真实问题描述。
- 对应案例名称。
- 处理链路说明。
- 可操作观察点。
- “打开案例”按钮。

卡片图片不应只做装饰。图片下方的文字要明确指出图中哪些元素用于学习：例如“窗外高光”“手机屏幕中的主体”“棋盘格标定板”“灰阶结构”“显示器色块”。

高级实验室的 before/after 预览也必须使用这些专用场景素材，而不是只使用抽象示意图：

- 对焦案例：同一场景图通过 blur 和焦平面标记表现镜头位置变化。
- 曝光案例：同一场景图通过 brightness、contrast 和高光 warning 表现 AE 取舍。
- 颜色案例：同一场景图通过 hue、saturation 和色偏蒙层表现白平衡/显示适配。
- 视频案例：同一场景图通过噪声层、拖影层和对比度变化表现降噪副作用。
- 计算摄影案例：同一场景图通过左右错位 ghost 和匹配网格表现双目/拼接置信度。
- AI 案例：同一场景图通过低清像素化、超分网格、去雾蒙层表现恢复强度和可信边界。

## 5. 测试要求

- `assets/scenarios/` 下六张图片必须存在。
- 设备场景数据不能再引用 `assets/demos/` 的旧 ISP 缩略图。
- 每个场景必须有 `pipeline` 和 `control` 字段。
- 点击每个场景按钮都能打开对应高级实验室。
- 高级实验室 before/after 必须引用 `assets/scenarios/`，并随滑条更新滤镜或叠层。
- 文档中必须说明医疗成像仅用于教学。

## 6. 后续扩展

后续可以把每个设备场景扩展成两层：

1. 场景卡片：告诉用户这类设备是什么、为什么难。
2. 案例组：同一设备下提供多个真实问题，例如手机摄影可以有逆光 HDR、夜景降噪、人像肤色、多摄融合；VR 可以有双目曝光一致性、畸变校正、低延迟运动补偿。
